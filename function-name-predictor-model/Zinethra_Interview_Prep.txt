2.1 — DATASET & PREPROCESSING
─────────────────────────────────────────

1. What challenges did you find in the new dataset compared to the original?

   Answer must cover:
   - Only 2 useful columns (description + keywords). Parameters, return_type,
     library were ALL empty. param_count column didn't exist at all.
   - Only 1 sample per function — impossible to do train/test split without
     augmentation (original had 3 variants per function)
   - Descriptions were heavily templated — 24 start_* functions all say
     "Begins recording a X workout session, tracking metrics like duration
     and calories." The ONLY difference is the sport name X.
   - 2 rows had corrupted Unicode (mojibake) in the Music control entries

2. How did you handle the corrupted Unicode characters?

   Answer:
   - Detected mojibake: "â â â¯Music" — UTF-8 bytes misinterpreted as Latin-1
   - Applied ASCII-only filter: keep only chars with ord 32-126
   - Collapsed multiple spaces with regex
   - Result: "Music" — clean and usable

3. How did you augment from 70 to 210 rows?

   Answer:
   - Created 2 additional description variants per function using
     intent-group-specific templates
   - Example for startYoga:
     Original:  "Begins recording a Yoga workout session..."
     Variant 1: "Initiates a yoga exercise session and starts tracking..."
     Variant 2: "Launches the yoga fitness activity to monitor..."
   - Keywords were shuffled per variant for TF-IDF variation
   - Used camel_tokenize() to convert "StrengthTraining" → "strength training"
     before inserting into templates
   - Result: 210 rows, 3 per class, enabling stratified 140/70 train/test split

4. Why didn't you just scrape more data from the internet?

   Answer:
   - Constraint: system must run 100% offline on embedded hardware
   - Real wearable SDK docs are proprietary — can't scrape them
   - Synthetic augmentation is controlled: we know exactly what each variant
     adds, no noisy web data that could confuse the model
   - 3 variants per class is enough for TF-IDF + linear model on this scale


─────────────────────────────────────────
2.2 — FEATURE ENGINEERING
─────────────────────────────────────────

5. Walk me through your build_feature_text() function.

   Answer:
   - Takes description + keywords only (no other columns available)
   - Keywords repeated 3x: f"{desc} {kw} {kw} {kw}"
   - Why 3x? Keywords contain the discriminative token (the sport name).
     In a sea of identical template words ("begins", "recording", "workout"),
     repeating "yoga yoga yoga" amplifies its TF-IDF weight above the noise.
   - All lowercased for consistent tokenization

6. Why two TF-IDF vectorizers instead of one?

   Answer:
   - Word TF-IDF (ngram_range=(1,2), max_features=2000): captures semantic
     tokens like "running", "heart rate", "remote control"
   - Char TF-IDF (analyzer='char_wb', ngram_range=(3,6), max_features=1000):
     captures sub-word patterns. "swimming" → trigrams "swi","wim","imm","min","ing"
   - Combined via scipy.sparse.hstack → 2035 total features
   - Result: word-only = 68% accuracy, word+char = 98% accuracy. The +30%
     improvement proves char n-grams are essential for this dataset.

7. Why did char n-grams help so much (+30%)?

   Answer:
   - The descriptions are 95% identical within each group. Word-level TF-IDF
     sees "begins recording workout session" in ALL 24 start_* rows —
     these words get LOW IDF scores (they appear everywhere)
   - The sport name ("running", "swimming") is the ONLY word that differs.
     But with word TF-IDF, it's just 1 token out of ~15. Low signal.
   - Char n-grams like "run", "unn", "nni", "nin" create MULTIPLE unique
     features from that single sport name, amplifying its weight
   - Also handles compound words: "strengthtraining" in keywords gets
     matched via "str","tre","ren","eng","ngt","gth","tht","htr","tra" etc.

8. What does sublinear_tf=True do?

   Answer:
   - Replaces raw term frequency with log(1 + tf)
   - Without it: a word appearing 6x (keyword 3x repeated × 2 mentions)
     gets 6x weight. With it: gets ~2.9x weight.
   - Prevents the 3x keyword repetition from completely dominating —
     we want boosted, not overwhelming


─────────────────────────────────────────
2.3 — HIERARCHICAL ROUTER (YOUR KEY UPGRADE)
─────────────────────────────────────────

9. Explain your hierarchical architecture.

   Answer:
   - Stage 1 — Intent Classifier: 6-class LogisticRegression
     Trained on all 140 training samples to predict intent:
     start_workout / stop_workout / live_monitor / get_data /
     enable_control / disable_control
     Test accuracy: 100%. This is easy — "begins/start" vs "ends/stop" vs
     "streams/live" are clearly distinct word patterns.

   - Stage 2 — Per-intent Target Classifiers: 6 separate LR models
     Each trained ONLY on its intent group's data.
     The start_workout classifier sees 48 training samples (24 functions × 2),
     and only needs to distinguish "running" from "swimming" from "yoga" etc.

   - At inference: Stage 1 predicts intent → routes to correct Stage 2 →
     Stage 2 predicts the specific function name

10. Why does each Stage 2 get its OWN TF-IDF vectorizer?

    Answer:
    - If you use a global vectorizer, IDF scores are computed across ALL 210 rows.
      "workout" appears in 96 rows (start + stop) → very low IDF globally.
    - With a group-local vectorizer for start_workout (48 rows), "workout"
      appears in ALL 48 → still low IDF. But "running" appears in only 2 →
      very HIGH IDF within the group. Exactly what we want.
    - The local IDF correctly identifies the sport name as the most important
      feature, rather than diluting it in the global vocabulary.

11. What happens if Stage 1 misclassifies the intent?

    Answer:
    - Stage 2 will predict from the WRONG group → guaranteed wrong output.
    - This is the cascading error risk of hierarchical classifiers.
    - Mitigation: Stage 1 is a 6-class problem with very distinct clusters.
      "Begins recording" vs "Ends the current" vs "Streams real-time" are
      trivially separable. Stage 1 achieved 100% test accuracy.
    - Additional safety: the OOD gate can catch cases where Stage 1 routes
      correctly but Stage 2 has low confidence.

12. Why not just use the flat classifier? It got 98.57% — isn't that enough?

    Answer:
    - 98.57% means 1 wrong prediction out of 70. On embedded hardware
      controlling real functions, 1 misfire is unacceptable.
    - The hierarchical approach gives us two confidence signals (intent + target)
      instead of one. Better for the OOD gate.
    - Architecturally, it shows I understand the data structure — not just
      throwing sklearn at the problem.
    - In production with 200+ functions, the flat classifier will degrade.
      Hierarchical scales — add new sports to start_workout without retraining
      get_data or enable_control classifiers.

13. How would you add a new function like "startTableTennis"?

    Answer:
    - Add 3 rows to the dataset (1 original + 2 augmented variants)
    - Retrain ONLY the start_workout Stage 2 sub-classifier
    - Stage 1 already knows "start_workout" intent — no retraining needed
    - Other Stage 2 classifiers untouched
    - This is a key advantage over flat: incremental updates, not full retrain


─────────────────────────────────────────
2.4 — OOD REJECT GATE (YOUR SAFETY LAYER)
─────────────────────────────────────────

14. Why do you need OOD detection on embedded hardware?

    Answer:
    - The model knows 70 functions. Users can say anything — "play spotify",
      "set alarm", "order food". Without OOD detection, the model WILL
      confidently return a wrong function. That's a hardware misfire.
    - The OOD gate is a safety valve: if the model is confused, return
      "command not recognized" instead of executing the wrong function.
    - Zero extra memory cost. Zero extra dependencies. Just an if-else
      on the probability distribution the model already computes.

15. Explain your two rejection signals.

    Answer:
    - Entropy: H(p) = -sum(p * log(p)). Measures overall confusion.
      Confident prediction: one class has 0.9, rest near 0 → low entropy.
      OOD input: probabilities spread evenly → high entropy.
      Threshold: 1.77 (reject if above).

    - Confidence Gap: top-1 probability minus top-2 probability.
      Confident: gap = 0.85 (clearly one best answer).
      Ambiguous: gap = 0.01 (two equally likely answers).
      Threshold: 0.07 (reject if below).

    - Using both gives dual safety: entropy catches "completely lost" inputs,
      gap catches "confused between two similar options" inputs.

16. How did you calibrate the thresholds?

    Answer:
    - Ran all 140 training samples through the hierarchical predictor
    - Collected entropy and gap values for CORRECTLY classified samples only
    - Set entropy threshold = max observed entropy × 2.0
    - Set gap threshold = min observed gap × 0.3
    - Conservative: allows ALL known correct predictions through, only
      rejects inputs that are clearly outside the observed distribution
    - Result: 0% false rejection, 100% OOD rejection

17. What are the failure modes of this approach?

    Answer:
    - An OOD input that happens to use keywords from the training data
      (e.g., "start the dishwasher" → might match start_workout with
      low but non-rejectable confidence). Mitigation: expand OOD test
      suite, possibly add a whitelist of valid activity names.
    - Threshold needs recalibration when new functions are added.
    - Top-3 entropy approximation is less accurate than full probability
      distribution entropy. For 24-class groups this is acceptable.

2.5 — DEPLOYMENT & CONSTRAINTS
─────────────────────────────────────────

18. How does your system meet all 4 constraints?

    Answer:
    - Memory (<5MB): Total model = 860KB. 6x under the limit.
    - Speed (sub-ms CPU): Flat = 0.1ms, Hierarchical = ~1ms. Both viable.
    - Offline: Zero internet. All inference is sklearn transform + predict.
      No API calls, no cloud, no downloads.
    - Semantic variance: Char n-grams handle typos/variants. Augmented
      descriptions provide diverse training phrasings. OOD gate handles
      completely unknown inputs.

19. How would you deploy this on an actual embedded chip?

    Answer:
    - Option A: Convert sklearn model to ONNX → run with ONNX Runtime
      (supports ARM, RISC-V, embedded Linux). Sub-MB model.
    - Option B: FastAPI wrapper on a companion phone/device that communicates
      with the wearable via BLE.
    - Option C: Reimplement the linear algebra in C/Rust — TF-IDF is just
      sparse matrix multiply, LR is just dot product + softmax. No sklearn
      dependency at inference time.
    - The 860KB pkl file is the Python-serialized version. ONNX export
      would be smaller.

20. Your inference latency is ~1ms. Is that fast enough?

    Answer:
    - Human voice command → text takes 200-500ms (speech-to-text latency).
      Adding 1ms for function prediction is negligible — user won't notice.
    - If they need <0.1ms: use the flat LR (word+char) model at 0.1ms.
      98.57% accuracy with no hierarchical routing.
    - Trade-off: 0.1ms + 98.57% vs 1ms + 100%. Present both, let them choose.

2.6 — "WHAT WOULD YOU IMPROVE?" QUESTIONS
─────────────────────────────────────────

21. What are the weaknesses of your current approach?

    Answer (be honest, then pivot to solutions):
    - Dataset is still small (210 augmented from 70). For production:
      collect real user voice command logs and retrain.
    - Augmentation templates are hand-written. Could use paraphrase models
      (T5, GPT) to generate diverse variants offline, then freeze for deployment.
    - The OOD gate uses top-3 entropy, not full distribution entropy.
      More accurate with full predict_proba, but requires storing 70-dim vector.
    - No per-domain confidence calibration — the start_workout group (24 classes)
      naturally has lower per-class confidence than live_monitor (4 classes).

22. If they gave you 500 real user commands, what would you change?

    Answer:
    - Replace synthetic augmentation with real data
    - Add a synonym expansion preprocessing layer (dict-based, zero memory cost)
    - Recalibrate OOD thresholds on real OOD examples
    - Possibly add a small FastText embedding layer (character-aware, handles
      OOV words better than TF-IDF) — still lightweight enough for embedded

23. Could you use a transformer model instead?

    Answer:
    - Not under the 5MB constraint. BERT-base is 440MB. Even DistilBERT is 250MB.
    - TinyBERT or MobileBERT: ~50-60MB. Still 10x over budget.
    - For this task (70 well-defined classes, structured input), TF-IDF + LR
      achieves 100% accuracy. A transformer cannot do better than 100%.
    - The right answer is: use the simplest model that solves the problem.
      Adding complexity for no accuracy gain is bad engineering.

SECTION 3 — RAPID FIRE DEFINITIONS (15 Terms)
════════════════════════════════════════════════════════════════════════════

Only terms directly relevant to YOUR project. Be able to define in 10 sec.

1.  TF-IDF         → Scores how unique a word is to a specific document
                     relative to all documents. High = distinctive, low = common.

2.  sublinear_tf   → log(1+tf) instead of raw count. Prevents high-frequency
                     terms from dominating the vector.

3.  char_wb        → TF-IDF analyzer that generates character n-grams only
                     within word boundaries. "swim" → "swi","wim".

4.  hstack         → scipy.sparse function that horizontally concatenates
                     sparse matrices. Used to fuse word + char features.

5.  Hierarchical   → Two-stage classification: coarse category first (intent),
    Classifier       then fine-grained prediction within that category.

6.  OOD Detection  → Identifying inputs that fall outside the model's training
                     distribution. Prevents confident wrong predictions.

7.  Shannon        → H(p) = -sum(p*log(p)). Measures uncertainty in a
    Entropy          probability distribution. High = confused model.

8.  Confidence Gap → Difference between top-1 and top-2 predicted probabilities.
                     Small gap = ambiguous prediction.

9.  LabelEncoder   → Converts string labels to integers and back.
                     fit on training, inverse_transform at inference.

10. class_weight   → 'balanced' adjusts loss weights inversely proportional to
                     class frequency. Prevents majority class domination.

11. predict_proba  → Returns probability distribution over all classes.
                     Enables top-k predictions and confidence scoring.

12. Stratified     → Train/test split that preserves class proportions.
    Split            With 3 per class: 2 train + 1 test guaranteed.

13. ShuffleSplit   → CV strategy with random non-stratified partitions.
                     Tests robustness across different data arrangements.

14. joblib         → Serialization library for sklearn objects. Faster than
                     pickle for numpy arrays. Used for model export.

15. mojibake       → Garbled text from encoding mismatch (UTF-8 read as Latin-1).
                     Your Music rows had this. Fixed with ASCII-only filter.
